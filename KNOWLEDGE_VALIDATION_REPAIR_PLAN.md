# 知识库回答校验链路修复方案

## 背景

当前知识库问答链路偏“严格事实问答”设计：检索不到候选、候选被判断为弱相关、模型自检为证据不足、引用校验失败，任一环节都会导致拒答。这可以降低幻觉，但在“基于简历生成面试问题 / 总结候选人优势 / 对项目经历做追问”这类文档生成与分析任务中，系统容易把可用证据误判为不足，导致用户明明导入了相关 PDF，仍收到拒答。

本方案从四个本质判断节点出发修复：

```text
1. 用户问题是否需要知识库？
2. 知识库是否召回了相关证据？
3. 模型回答是否只基于这些证据？
4. 回答中的每个引用是否真的有效？
```

目标不是降低可信度，而是把“拒答”从单一硬门槛改为更精确的证据状态判断，使系统能区分：

- 完全无关，应该拒答。
- 证据不足以回答事实问题，应该拒答或澄清。
- 证据足以支持总结、改写、追问、面试题生成等任务，应该回答并标注依据。
- 回答中存在局部引用缺陷，应该重试或降级，而不是直接整体失败。

## 当前问题诊断

### 1. 意图层：把生成/分析任务误当成事实问答

用户提问如：

```text
假设你是面试官，基于我的简历对我提问
总结这份简历的项目亮点
帮我根据简历准备自我介绍
```

这些问题确实需要知识库，但不是单纯的“文档里写了什么”。它们属于“基于知识库的生成/分析/转换任务”。

当前链路更接近：

```text
knowledge_fact_qa
```

缺少：

```text
knowledge_summary
knowledge_transform
knowledge_interview
knowledge_review
```

结果是模型被要求“每个回答段落必须有 evidence_chunk_ids”，导致组织句、建议句、生成的问题也被当成必须逐段引用的事实陈述。

### 2. 检索层：弱相关拒答对短任务指令过于敏感

当前低相关判断主要保护“天气？”这类无关问题，但简历任务常见查询非常短，例如：

```text
模拟面试
根据简历提问
评价我的项目
```

这类 query 与简历原文可能只有少量词面重叠，关键词召回弱，主要依赖语义检索。若系统看到“没有关键词命中 + 仅语义命中 + query/chunk 文本交集弱”，就可能触发弱相关拒答。

这会把“任务表达和文档表达不一致”误判为“知识库无关”。

### 3. 生成层：证据充分性只有布尔值，无法表达部分充分

当前模型输出只用：

```text
evidence_sufficient: true | false
```

但真实场景至少有三种状态：

```text
strong_evidence   证据足够直接回答
partial_evidence  证据足够支持部分回答或生成任务
no_evidence       证据不足，应拒答
```

简历任务经常是 `partial_evidence`：资料足以生成面试问题，但不一定足以回答所有推断性问题。当前布尔值会迫使模型在 true/false 之间硬选，偏安全时就会 false。

### 4. 引用层：段落级强制引用过于粗糙

当前校验是“每个段落至少一个有效证据”。这对事实段落合理，但对以下段落不合理：

```text
下面我会从项目复杂度、技术深度和协作能力三个方向提问。
你可以优先准备以下追问。
如果需要，我可以继续模拟一轮面试。
```

这些是组织句或交互引导，不是事实主张。它们不应因为缺少引用导致整段回答无效。

## 修复目标

### 功能目标

- 系统能识别用户问题属于事实问答、总结、生成、面试模拟、改写、评价等不同知识库任务。
- 对知识库相关但非事实型任务，不再因“每段都必须引用”而过早拒答。
- 对明显无关问题仍然拒答。
- 对引用合法性仍保持严格：引用 chunk 必须存在、来自当前检索候选、属于当前索引、来源可定位。
- 拒答时能返回明确原因，便于用户和开发者判断是哪一层失败。

### 非目标

- 不允许模型脱离知识库编造事实。
- 不取消引用校验。
- 不引入完全开放的闲聊回答路径。
- 不把系统元信息、运行工具问题伪装成知识库引用。

## 新链路设计

```text
用户问题
  ↓
1. 意图分流
  ├─ system_meta：系统元信息，直接回答，不需要知识库引用
  ├─ knowledge_fact_qa：事实问答，强证据强引用
  ├─ knowledge_summary：总结提炼，事实句引用，组织句可无引用
  ├─ knowledge_transform：改写/生成/面试题，依据引用，生成句弱引用
  └─ ambiguous：低置信度时走普通知识库问答或澄清
  ↓
2. 检索与证据状态评估
  ├─ no_candidates：拒答
  ├─ weak_candidates：根据意图决定拒答、澄清或谨慎回答
  ├─ partial_evidence：允许受限回答
  └─ strong_evidence：允许正常回答
  ↓
3. 生成约束
  ├─ 只能基于候选证据陈述事实
  ├─ 生成型内容必须说明依据来自哪些候选证据
  └─ 不确定内容必须标记为无法判断
  ↓
4. 引用校验
  ├─ fact 段落/事实句：必须有有效引用
  ├─ generated 段落：至少要能追溯到依据 chunk
  ├─ transition 段落：允许无引用
  └─ 引用 chunk 合法性仍严格校验
```

## 具体修复方案

### P0：增加知识库任务意图分流

新增或扩展意图类型：

```text
knowledge_fact_qa
knowledge_summary
knowledge_transform
knowledge_interview
knowledge_review
```

建议规则：

- 包含“是什么、谁、什么时候、列出、根据原文、资料里是否提到”等，进入 `knowledge_fact_qa`。
- 包含“总结、概括、提炼、亮点、优势、不足”等，进入 `knowledge_summary`。
- 包含“改写、润色、生成、写一段、自我介绍、邮件、简历描述”等，进入 `knowledge_transform`。
- 包含“面试、追问、面试官、模拟、准备问题”等，进入 `knowledge_interview`。
- 包含“评价、分析、建议、优化、如何改进”等，进入 `knowledge_review`。

所有知识库任务仍然必须先检索；区别在于证据充分性和引用策略不同。

### P0：把证据状态从布尔值升级为枚举

建议新增内部字段：

```ts
evidenceStatus:
  | "strong_evidence"
  | "partial_evidence"
  | "weak_evidence"
  | "no_evidence"
```

映射策略：

```text
no_candidates → no_evidence
明显无关 → weak_evidence
有候选但不足以回答事实问题 → partial_evidence
候选能支撑主要事实 → strong_evidence
```

对于不同意图：

| 意图 | no_evidence | weak_evidence | partial_evidence | strong_evidence |
| --- | --- | --- | --- | --- |
| fact_qa | 拒答 | 拒答 | 谨慎回答或澄清 | 回答 |
| summary | 拒答 | 谨慎摘要或澄清 | 回答并提示范围 | 回答 |
| transform | 拒答 | 澄清 | 回答并标注依据 | 回答 |
| interview | 拒答 | 澄清 | 生成问题并引用依据 | 回答 |
| review | 拒答 | 谨慎建议 | 回答并区分事实/建议 | 回答 |

### P0：调整低相关拒答策略

当前 `low_relevance_candidates` 应保留，但不能一刀切。

建议：

1. 对明显无关短问仍拒答：

```text
天气？
今天几号？
讲个笑话？
```

2. 对文档任务词命中时不要直接拒答：

```text
简历、面试、项目、经历、技能、候选人、总结、提问、评价、改写
```

3. 低相关时返回 `weak_evidence`，由意图层决定：

```text
fact_qa + weak_evidence → 拒答
interview + weak_evidence → 要求用户明确范围或选择资料
summary + weak_evidence → 提示“检索到的资料较少”，谨慎摘要
```

### P1：改造生成 Prompt

事实问答 prompt：

```text
只回答候选证据直接支持的事实。每个事实段落必须引用 evidence_chunk_ids。
```

总结/分析 prompt：

```text
你可以总结和归纳候选证据，但不得引入候选证据之外的新事实。
事实性结论必须引用来源；组织性语言不需要引用。
如果只能部分支持，请明确说明“以下基于已检索到的资料片段”。
```

面试/生成 prompt：

```text
请基于候选证据生成面试问题。每个问题应能追溯到至少一个候选 chunk。
不要声称简历中没有出现的经历。可以提出追问，但必须说明追问依据。
```

建议输出结构增加段落类型：

```json
{
  "paragraphs": [
    {
      "type": "fact",
      "text": "...",
      "evidence_chunk_ids": ["..."]
    },
    {
      "type": "generated_question",
      "text": "...",
      "evidence_chunk_ids": ["..."]
    },
    {
      "type": "transition",
      "text": "...",
      "evidence_chunk_ids": []
    }
  ]
}
```

### P1：引用校验按段落类型区分

继续严格校验引用本身是否合法：

- chunk 必须存在；
- chunk 必须来自本次检索候选；
- chunk 必须属于当前 index version；
- index/source/source_version 必须 ready；
- 外部资料必须确认并索引；
- chunk 必须可定位。

但“是否必须有引用”应按段落类型决定：

| 段落类型 | 是否必须引用 |
| --- | --- |
| fact | 必须 |
| summary_claim | 必须 |
| generated_question | 必须至少有依据 chunk |
| recommendation | 建议必须引用依据，泛化操作建议可无引用 |
| transition | 不强制 |
| clarification | 不强制 |

### P1：拒答诊断信息标准化

建议响应中增加：

```ts
refusalDiagnostics?: {
  stage:
    | "intent_routing"
    | "retrieval"
    | "relevance_gate"
    | "model_self_check"
    | "citation_validation";
  reason: string;
  evidenceStatus?: string;
  candidateCount?: number;
  invalidCitationReasons?: string[];
}
```

前端展示建议：

```text
┌────────────────────────────────────────────┐
│ 证据不足，已拒答                            │
│ 阶段：引用校验                              │
│ 原因：第 2 段没有有效引用                    │
│ 候选片段：6 个                               │
│ 建议：可以改问“基于简历生成面试问题”          │
└────────────────────────────────────────────┘
```

### P2：检索增强

针对简历类 PDF，建议增加查询扩展：

```text
面试问题 → 项目经历、技术栈、职责、成果、难点、协作、实习、教育经历
评价简历 → 项目、技能、成果、职责、亮点、不足
自我介绍 → 教育、项目、技术栈、经历、成果
```

对于 `knowledge_interview`，可以把用户 query 扩展成：

```text
用户原问题 + 项目经历 + 技术栈 + 工作职责 + 项目成果 + 难点
```

这样能提高关键词召回，减少只靠 semantic 召回导致的误拒答。

### P2：增加二阶段证据评估

检索后增加一个轻量证据评估器：

```text
输入：query、intent、top chunks
输出：evidenceStatus、supportingChunks、missingInfo
```

该评估器只判断：

- 候选是否主题相关；
- 能支持哪些子问题；
- 不能支持哪些部分。

它不直接生成最终回答。

## 测试计划

### 必须通过的正向用例

1. 简历面试题生成：

```text
基于我的简历，假设你是面试官，对我提问
```

预期：

- 不拒答；
- 生成多条面试问题；
- 每个问题至少引用一个简历 chunk；
- 回答不声称简历中不存在的经历。

2. 简历总结：

```text
总结这份简历的项目亮点
```

预期：

- 输出分段总结；
- 事实性亮点带引用；
- 不要求过渡句带引用。

3. 简历评价：

```text
从面试官角度评价这份简历的优势和风险
```

预期：

- 区分“简历中可见事实”和“基于事实的建议”；
- 事实引用必须有效；
- 建议不能虚构经历。

### 必须拒答的负向用例

1. 完全无关问题：

```text
今天上海天气怎么样？
```

预期：

- 拒答；
- reason 为 `no_evidence` 或 `weak_evidence`。

2. 简历中不存在的具体事实：

```text
候选人在字节跳动做过什么？
```

如果简历没有字节跳动经历，预期：

- 拒答或说明资料未提到；
- 不编造。

3. 引用伪造：

模型返回不在候选中的 chunk id。

预期：

- citation validation fail；
- 重试一次；
- 仍失败则拒答并记录 invalid reason。

## 验收标准

- 简历 PDF 的面试题生成、总结、评价类问题不再被默认拒答。
- 明显无关问题仍会拒答。
- 所有事实性内容仍带有效引用。
- 引用 chunk 合法性校验保持严格。
- 拒答结果能说明失败阶段和原因。
- Agent Run Trace 能显示：意图分流、检索数量、证据状态、引用校验结果。

## 推荐实施顺序

1. 扩展知识库意图类型，先识别 `knowledge_interview`、`knowledge_summary`、`knowledge_transform`。
2. 增加 `evidenceStatus`，把低相关拒答改为证据状态输出。
3. 按意图拆分 prompt。
4. 引用校验器支持段落类型，允许 transition/clarification 无引用。
5. 增加拒答诊断字段。
6. 增加简历 PDF 场景回归测试。

## 风险与边界

- 放宽拒答不等于放宽事实引用。事实引用仍应保持强校验。
- 生成型任务要避免“看起来合理但资料未支持”的虚构问题。
- 若检索召回质量差，放宽校验只能缓解误拒答，不能替代检索增强。
- 如果用户要求高度事实性的判断，例如“他是否有 X 经历”，仍应采用事实问答的严格模式。
