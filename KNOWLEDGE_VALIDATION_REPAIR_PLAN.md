# 知识库回答校验链路泛化修复方案

## 背景

当前知识库问答链路偏“严格事实问答”设计：检索不到候选、候选被判断为弱相关、模型自检为证据不足、引用校验失败，任一环节都可能导致拒答。这能降低幻觉，但当用户提出“基于资料做总结、改写、生成问题、评价建议、提炼结构”等任务时，系统容易把可用证据误判为不足。

此前测试中使用“简历 PDF”暴露了问题，但修复目标不能绑定简历场景。相同问题也会出现在：

- 产品文档：基于接口说明生成接入步骤、FAQ、风险清单。
- 会议纪要：总结决议、提炼待办、生成追问。
- 论文或报告：概括贡献、分析方法优缺点、生成阅读问题。
- 合同或制度：提炼义务、生成审查清单、指出风险点。
- 项目资料：总结进展、生成复盘问题、改写汇报稿。
- 简历资料：生成面试问题、总结项目亮点、准备自我介绍。

因此，本方案按“任务形态 + 证据状态 + 引用责任”修复，而不是按具体资料类型修复。

## 本质判断链路

所有知识库回答都先经过四个判断节点：

```text
1. 用户问题是否需要知识库？
2. 知识库是否召回了相关证据？
3. 模型回答是否只基于这些证据？
4. 回答中的每个引用是否真的有效？
```

修复目标不是降低可信度，而是把“拒答”从单一硬门槛改为分层判断，使系统能区分：

- 完全无关：应该拒答。
- 事实问题证据不足：应该拒答或澄清。
- 证据能支持总结、改写、问题生成、评价建议：应该受限回答并标注依据。
- 局部引用缺陷：应该重试、降级或局部提示，而不是无差别整体失败。

## 当前问题诊断

### 1. 意图层：把资料处理任务误当成事实问答

用户可能这样问：

```text
根据这份接口文档生成接入步骤
总结这份会议纪要的关键决议
基于论文内容设计 5 个阅读讨论问题
从合同条款中提炼乙方主要义务
基于我的简历模拟面试追问
```

这些问题都需要知识库，但不是单纯的“文档里写了什么”。它们属于“基于知识库的资料处理任务”。

当前链路如果统一当作：

```text
knowledge_fact_qa
```

就会把组织句、建议句、生成的问题、结构化标题都当成必须逐段引用的事实陈述，导致模型更容易选择拒答。

### 2. 检索层：短任务指令容易触发弱相关误判

低相关判断主要用于保护无关问题，例如：

```text
天气？
今天几号？
讲个笑话？
```

但很多合法的知识库任务指令也很短：

```text
总结一下
生成问题
提炼风险
改写成汇报
模拟追问
```

这些 query 与资料原文可能没有明显词面重叠，主要依赖语义检索。若系统看到“没有关键词命中 + 仅语义命中 + query/chunk 文本交集弱”，就可能把“任务表达和资料表达不一致”误判为“知识库无关”。

### 3. 生成层：证据充分性只有布尔值，无法表达部分充分

当前模型输出只用：

```text
evidence_sufficient: true | false
```

但真实场景至少有四种状态：

```text
strong_evidence   证据足够直接回答主要问题
partial_evidence  证据可支持部分回答或资料处理任务
weak_evidence     候选相关性弱，需要更严格限制回答范围
no_evidence       没有可用证据，应拒答
```

资料处理任务经常是 `partial_evidence` 或 `weak_evidence`：资料足以支持“基于已检索片段的总结/问题生成/建议”，但不足以支持开放式推断。布尔值会迫使模型在 true/false 之间硬选，偏安全时就会误拒答。

### 4. 引用层：段落级强制引用过于粗糙

当前校验接近“每个段落至少一个有效证据”。这对事实段落合理，但对以下内容不完全合理：

```text
下面从背景、风险和下一步三个角度说明。
你可以优先关注以下问题。
如果需要，我可以继续把它改写成邮件版本。
```

这些是组织句、过渡句或交互引导，不是事实主张。它们不应因为缺少引用导致整段回答无效。

## 修复目标

### 功能目标

- 系统能识别用户问题属于事实问答、总结提炼、转换改写、问题生成、评价建议等不同知识库任务。
- 对知识库相关但非事实型任务，不再因“每段都必须引用”或“短 query 低相关”而过早拒答。
- 对明显无关问题仍然拒答。
- 对引用合法性保持严格：引用 chunk 必须存在、来自当前检索候选、属于当前索引、来源可定位。
- 拒答时返回明确原因，便于判断失败发生在意图、检索、证据状态、模型自检还是引用校验。

### 非目标

- 不允许模型脱离知识库编造事实。
- 不取消引用校验。
- 不引入完全开放的闲聊回答路径。
- 不把系统元信息、运行工具问题伪装成知识库引用。
- 不为某一类资料定制不可泛化的规则。

## 泛化链路设计

```text
用户问题
  ↓
1. 意图分流
  ├─ system_meta：系统元信息，直接回答，不需要知识库引用
  ├─ knowledge_fact_qa：事实问答，强证据强引用
  ├─ knowledge_summary：总结提炼，事实性结论引用，组织句可无引用
  ├─ knowledge_transform：改写/生成，来源事实引用，表达转换可无新增事实
  ├─ knowledge_question_generation：基于资料生成问题/追问/清单，每个问题可追溯依据
  ├─ knowledge_review：评价/建议/风险分析，区分资料事实与基于事实的建议
  └─ knowledge_ambiguous：低置信度时走保守问答或请求澄清
  ↓
2. 检索与证据状态评估
  ├─ no_candidates：拒答
  ├─ weak_candidates：根据意图决定拒答、澄清或受限回答
  ├─ partial_evidence：允许限定范围回答
  └─ strong_evidence：允许正常回答
  ↓
3. 生成约束
  ├─ 事实只能来自候选证据
  ├─ 资料处理结果必须说明依据或可追溯到候选证据
  ├─ 证据不足的部分必须标记为无法判断或请求补充资料
  └─ 不得把通用常识伪装成资料事实
  ↓
4. 引用校验
  ├─ fact / claim：必须有有效引用
  ├─ summary_claim：必须有有效引用
  ├─ generated_question / recommendation：必须可追溯到依据 chunk
  ├─ transition / clarification：允许无引用
  └─ 引用 chunk 合法性始终严格校验
```

## 具体修复方案

### P0：增加知识库任务意图分流（✅ 已修复）

建议意图类型：

```text
knowledge_fact_qa
knowledge_summary
knowledge_transform
knowledge_question_generation
knowledge_review
knowledge_ambiguous
```

原 `knowledge_interview` 已并入 `knowledge_question_generation`，面试追问只是问题生成的一种领域实例。

建议规则：

- `knowledge_fact_qa`
  - 触发词：是什么、谁、什么时候、是否提到、列出、根据原文、资料里写了什么。
  - 约束：只回答候选证据直接支持的事实。

- `knowledge_summary`
  - 触发词：总结、概括、提炼、归纳、亮点、要点、关键结论、核心内容。
  - 约束：可归纳结构，但事实性结论必须引用。

- `knowledge_transform`
  - 触发词：改写、润色、生成、写一段、整理成、转成、压缩成、扩写成、汇报稿、邮件、说明文案。
  - 约束：只能转换表达，不得新增资料外事实。

- `knowledge_question_generation`
  - 触发词：提问、追问、模拟、准备问题、检查清单、审查清单、讨论问题、FAQ、测试问题。
  - 约束：每个问题或清单项必须可追溯到至少一个候选 chunk。

- `knowledge_review`
  - 触发词：评价、分析、建议、优化、风险、短板、改进方向、优缺点、可行性。
  - 约束：区分“资料中可见事实”和“基于事实的建议/判断”。

- `knowledge_ambiguous`
  - 触发词：这份、这篇、资料、文档、文件、报告、论文、合同、会议纪要、接口文档、项目材料等资料指代词，但没有明确任务或事实问法。
  - 约束：走知识库检索，但回答必须保守；无法判断用户要做什么时请求澄清。

所有知识库任务仍然必须先检索；区别在于证据充分性和引用责任不同。

### P0：把证据状态从布尔值升级为枚举（✅ 已修复）

建议新增内部字段：

```ts
evidenceStatus:
  | "strong_evidence"
  | "partial_evidence"
  | "weak_evidence"
  | "no_evidence"
```

映射策略：

```text
no_candidates → no_evidence
明显无关 → weak_evidence
候选片段主题相关但覆盖不完整 → partial_evidence
候选片段能支撑主要事实或主要任务 → strong_evidence
```

对于不同意图：

| 意图 | no_evidence | weak_evidence | partial_evidence | strong_evidence |
| --- | --- | --- | --- | --- |
| fact_qa | 拒答 | 拒答 | 谨慎回答或澄清 | 回答 |
| summary | 拒答 | 说明范围后谨慎摘要 | 回答并提示范围 | 回答 |
| transform | 拒答 | 说明范围后受限生成 | 回答并标注依据 | 回答 |
| question_generation | 拒答 | 生成少量可追溯问题或澄清 | 生成问题并引用依据 | 回答 |
| review | 拒答 | 谨慎建议或澄清 | 回答并区分事实/建议 | 回答 |

### P0：调整低相关拒答策略（✅ 已修复）

`low_relevance_candidates` 应保留，但不能一刀切。

建议：

1. 对明显无关短问仍拒答：

```text
天气？
今天几号？
讲个笑话？
股票涨了吗？
```

2. 对通用资料任务词命中时不要直接拒答：

```text
总结、概括、提炼、改写、润色、生成、提问、追问、清单、评价、分析、建议、风险、优化
```

3. 对资料指代词命中时不要仅凭短 query 拒答：

```text
这份、这篇、资料、文档、文件、报告、论文、合同、会议纪要、说明书、项目材料、PDF
```

4. 低相关时返回 `weak_evidence`，由意图层决定：

```text
fact_qa + weak_evidence → 拒答或澄清
summary + weak_evidence → 提示“仅基于已检索片段”，谨慎摘要
transform + weak_evidence → 只转换候选片段中可见信息
question_generation + weak_evidence → 只生成可追溯问题，必要时要求用户明确范围
review + weak_evidence → 只给出基于片段的有限建议，不能泛化结论
```

### P1：改造生成 Prompt（✅ 已修复）

事实问答 prompt：

```text
只回答候选证据直接支持的事实。每个事实段落必须引用 evidence_chunk_ids。
证据不足时必须拒答或说明资料未提到。
```

总结/分析 prompt：

```text
你可以总结和归纳候选证据，但不得引入候选证据之外的新事实。
事实性结论必须引用来源；结构性组织语言不需要引用。
如果只能部分支持，请明确说明“以下仅基于已检索到的资料片段”。
```

转换/改写 prompt：

```text
请基于候选证据改写或生成目标文本。
可以改变表达形式，但不得新增候选证据之外的人物、项目、数据、结论或承诺。
关键事实必须保留引用依据。
```

问题生成 prompt：

```text
请基于候选证据生成问题、追问、FAQ、检查清单或讨论题。
每个问题必须能追溯到至少一个候选 chunk。
不要围绕资料中不存在的信息发问，除非明确标记为“需要补充资料确认”。
```

评价建议 prompt：

```text
请基于候选证据给出评价、风险或建议。
必须区分“资料事实”和“基于事实的建议”。
不得把建议写成资料中已经发生或已经确认的事实。
```

结构化回答输出已增加段落类型：

> 当前实现要求结构化模型逐段输出 `type`，并在归一化、自动分段和引用校验过程中保留该字段。纯文本 fallback 无法可靠表达类型，因此仍按 `fact` 严格校验。

```json
{
  "paragraphs": [
    {
      "type": "fact",
      "text": "...",
      "evidence_chunk_ids": ["..."]
    },
    {
      "type": "generated_question",
      "text": "...",
      "evidence_chunk_ids": ["..."]
    },
    {
      "type": "transition",
      "text": "...",
      "evidence_chunk_ids": []
    }
  ]
}
```

### P1：引用校验按段落类型区分（✅ 已修复）

继续严格校验引用本身是否合法：

- chunk 必须存在；
- chunk 必须来自本次检索候选；
- chunk 必须属于当前 index version；
- index/source/source_version 必须 ready；
- 外部资料必须确认并索引；
- chunk 必须可定位。

“是否必须有引用”已按段落类型决定：

| 段落类型 | 是否必须引用 |
| --- | --- |
| fact | 必须 |
| summary_claim | 必须 |
| generated_question | 必须至少有依据 chunk |
| recommendation | 必须；纯操作提示应标注为 transition 或 clarification |
| transformed_text | 必须 |
| transition | 不强制 |
| clarification | 不强制 |

实现边界：

- `transition` / `clarification` 没有引用时可以通过；一旦提供引用，引用本身仍执行完整合法性校验。
- 缺失段落类型的旧数据按 `fact` 处理，保持原有严格行为。
- 未知段落类型返回 `paragraph_type_not_supported`，不能借自定义类型绕过引用责任。
- 校验结果逐段返回 `type` 和 `citationRequired`，供响应、持久化与诊断使用。

```text
│ 原因：第 2 段没有有效引用                    │
│ 候选片段：6 个                               │
│ 建议：请指定要分析的资料范围，或补充相关文档  │
└────────────────────────────────────────────┘
```

### P2：泛化检索增强（⬜ 未修复）

检索增强不应写死某一资料类型，而应按任务形态扩展查询：

```text
summary → 主题、要点、结论、背景、数据、结果、问题、行动项
transform → 原始表述、关键事实、对象、时间、数据、约束、目标格式
question_generation → 背景、关键事实、流程、风险点、决策依据、未说明信息
review → 优点、缺点、风险、约束、证据、结果、改进建议、影响
fact_qa → 用户问题中的实体、属性、时间、地点、数值、原文表述
```

可以针对资料类型增加轻量插件式扩展，但必须保持“任务形态优先，资料类型辅助”：

```text
合同/制度 → 条款、义务、责任、期限、违约、审批、风险
论文/报告 → 方法、实验、数据、结论、限制、贡献
会议纪要 → 决议、行动项、负责人、时间、风险、阻塞
产品/接口文档 → 参数、流程、限制、错误码、示例、依赖
简历/项目资料 → 经历、职责、成果、技术、难点、协作
```

### P2：增加二阶段证据评估（⬜ 未修复）

检索后增加一个轻量证据评估器：

```text
输入：query、intent、top chunks
输出：evidenceStatus、supportingChunks、missingInfo、answerableScope
```

该评估器只判断：

- 候选是否主题相关；
- 能支持哪些子问题；
- 不能支持哪些部分；
- 哪些回答范围必须被限制。

它不直接生成最终回答。

## 测试计划

### 必须通过的正向用例

1. 产品文档任务生成：

```text
根据这份接口文档生成接入步骤和注意事项
```

预期：

- 不拒答；
- 步骤中的关键事实带引用；
- 不补充文档外参数、错误码或承诺。

2. 会议纪要总结：

```text
总结这份会议纪要的关键决议和待办
```

预期：

- 输出分段总结；
- 决议、负责人、时间等事实带引用；
- 过渡句不要求引用。

3. 论文阅读问题生成：

```text
基于这篇论文生成 5 个讨论问题
```

预期：

- 不拒答；
- 每个问题能追溯到论文片段；
- 不围绕论文未提到的方法或实验发问。

4. 合同风险审查：

```text
从这份合同中提炼主要风险点
```

预期：

- 区分合同原文事实和风险判断；
- 风险判断必须能回溯到相关条款；
- 不引入合同外义务。

5. 简历场景回归：

```text
基于我的简历生成面试追问
```

预期：

- 简历作为普通资料类型参与测试；
- 生成问题必须可追溯到简历片段；
- 不为简历场景设置特殊豁免。

### 必须拒答或澄清的负向用例

1. 完全无关问题：

```text
今天上海天气怎么样？
```

预期：

- 拒答；
- reason 为 `no_evidence` 或 `weak_evidence`。

2. 资料中不存在的具体事实：

```text
这份合同是否承诺 30 天无条件退款？
```

如果候选证据没有该条款，预期：

- 拒答或说明资料未提到；
- 不编造。

3. 引用伪造：

模型返回不在候选中的 chunk id。

预期：

- citation validation fail；
- 重试一次；
- 仍失败则拒答并记录 invalid reason。

4. 弱证据过度泛化：

```text
根据检索到的一小段内容，评价整个项目是否成功
```

如果候选只覆盖局部信息，预期：

- 只能说明局部可见事实；
- 必须提示无法判断整体成功与否。

## 验收标准

- 产品文档、会议纪要、论文、合同、项目材料、简历等不同资料类型的资料处理任务都不因默认事实问答规则误拒答。
- 明显无关问题仍会拒答。
- 事实性内容仍带有效引用。
- 生成、总结、评价等内容能追溯到候选证据。
- 引用 chunk 合法性校验保持严格。
- 拒答结果能说明失败阶段和原因。
- Agent Run Trace 能显示：意图分流、检索数量、证据状态、引用校验结果。

## 风险与边界

- 放宽拒答不等于放宽事实引用。事实引用仍应保持强校验。
- 资料处理任务要避免“看起来合理但资料未支持”的虚构内容。
- 若检索召回质量差，放宽校验只能缓解误拒答，不能替代检索增强。
- 如果用户要求高度事实性的判断，例如“资料是否明确提到 X”，仍应采用事实问答的严格模式。
- 任何资料类型都不能拥有绕过证据校验的特权；资料类型只能影响检索扩展词，不能改变可信边界。
