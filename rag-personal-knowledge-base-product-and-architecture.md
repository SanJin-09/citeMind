# 基于 RAG 的个人知识库助手：产品需求与技术架构方案

> 文档状态：初始产品与架构基线
> 更新日期：2026-06-07

## 1. 产品定位

### 1.1 产品定义

本产品是一款面向学生、咨询顾问和自媒体创作者的桌面端知识复习总结与写作助手。

用户通过自己的 Seed API（火山方舟 Ark API Key），将散落的 Word 文档、PDF 文件、图片和网页链接转化为可检索、可提问、可总结、可引用的个人知识库，并基于知识库生成大纲、初稿和总结。

产品强调以下核心价值：

- **本地优先**：原始资料、解析结果、向量索引、对话历史和 API 配置默认保存在本地。
- **用户自带 Seed API Key**：首版固定接入字节跳动 Seed 模型，用户提供自己的火山方舟 Ark API Key。
- **证据可追溯**：回答中的每段内容展示来源、页码、引用原文和相关程度。
- **辅助产出**：不仅回答问题，还帮助用户完成复习总结、资料对比、写作大纲和初稿。
- **知识持续维护**：检测重复资料、网页更新和过期资料，并自动归类、自动关联。

### 1.2 目标用户

| 用户群体 | 主要需求 |
|---|---|
| 学生 | 复习课程资料、总结论文、比较观点、生成学习提纲 |
| 咨询顾问 | 快速理解项目资料、提取证据、生成报告大纲与初稿 |
| 自媒体创作者 | 整理选题资料、关联素材、生成有引用依据的文章 |

### 1.3 产品边界

首版聚焦个人使用，不包含以下能力：

- 多人协作和复杂权限系统
- 本地大语言模型推理
- 移动端应用
- 插件市场
- 完整笔记编辑器
- 复杂知识图谱编辑器
- 应用关闭后的常驻网页更新服务

说明：首版不支持本地大语言模型，但文档解析、OCR、向量存储等能力可以在本地运行。

## 2. 核心使用场景

### 2.1 知识导入与维护

用户可以：

- 导入 PDF、DOCX、图片和网页链接。
- 查看文件解析、OCR 和索引进度。
- 检查解析结果与失败原因。
- 自动识别完全重复和内容重复的文件。
- 检测已导入网页的正文更新。
- 标记可能过时、已被替代或内容冲突的资料。
- 根据主题自动归类资料。
- 自动发现资料、人物、主题和观点之间的关联。

### 2.2 知识检索与问答

用户可以：

- 通过关键词或自然语言搜索资料。
- 询问资料中的事实和细节。
- 比较不同文档中的观点。
- 按主题、人物或时间线汇总信息。
- 查看回答所使用的来源、页码和引用原文。
- 点击引用跳转至原始文档位置。

### 2.3 复习与写作

用户可以：

- 生成知识点总结和复习提纲。
- 根据资料生成文章或报告大纲。
- 根据大纲逐节生成初稿。
- 为每段内容匹配资料证据。
- 检查内容是否缺少证据或与资料冲突。
- 将结果导出为 Markdown，后续可扩展 Word 导出。

## 3. 核心产品原则

### 3.1 证据优先

模型只能引用本次检索得到且经过后端校验的文本块。模型不得自行生成来源、页码或引用编号。

当证据不足时，系统应明确说明知识库中没有足够信息，而不是让模型自由补全事实。

### 3.2 模型与知识库解耦

知识库、文档版本、文本块、引用和索引版本均使用产品自己的数据结构。LangChain 仅用于模型调用和流程编排，不作为业务数据的唯一保存形式。

### 3.3 后台任务可恢复

文档解析、OCR、Embedding、网页更新和索引重建均为可持久化后台任务。应用异常退出后，应能够继续或重新执行任务。

### 3.4 用户可控制

自动分类、自动关联和过期判断属于辅助能力。系统应展示判断依据，并允许用户确认、修改或忽略。

## 4. 首版功能范围

### 4.1 MVP 必须实现

1. 创建和管理多个知识库。
2. 导入 PDF、DOCX、图片和网页链接。
3. 展示解析、OCR、Embedding 和索引状态。
4. 支持关键词检索与向量检索。
5. 基于知识库进行问答。
6. 每段回答展示来源、页码、引用原文和相关程度。
7. 点击引用跳转至原文位置。
8. 配置火山方舟 Ark API Key。
9. 展示首版支持的 Seed 模型，并验证当前 API Key 对各模型的可调用状态。
10. 支持切换对话模型。
11. 支持 Embedding 模型索引版本化重建。
12. 检测完全重复文件和内容重复文件。
13. 默认在本地保存所有知识库资产和对话数据。
14. 支持 Markdown 导出。

### 4.2 第二阶段功能

- 网页内容更新检测与版本对比
- 过期资料标记
- 自动分类
- 自动关联
- 近似重复资料检测
- 大纲到长文的多步骤写作工作流
- 引用检查与自动修订
- Word 导出

| 模型调用 | LangChain Core + 火山方舟 Ark SDK | Seed 模型调用、流式输出和结构化输出 |
| 复杂工作流 | LangGraph，第二阶段引入 | 长文写作、引用检查和失败恢复 |
| 应用打包 | electron-builder | 跨平台打包和发布 |

### 5.2 LangChain 的使用边界

适合交给 LangChain：

- Seed 模型调用流程封装
- Prompt Template
- 流式输出
- 结构化输出
- 超时、重试和 Token 使用量统计

不应交给 LangChain：

- 文档与网页版本管理
- 重复文件检测
- Embedding 索引版本管理
- 网页更新检测
- 自动分类和自动关联数据
- 引用真实性验证
- SQLite 业务数据模型
- 后台任务状态和恢复

建议定义独立的模型网关接口，首版由火山方舟 Ark SDK 实现具体模型调用，LangChain Core 负责上层流程封装：

```python
class ModelGateway(Protocol):
    async def validate_model(self, model_id: str, role: ModelRole) -> ModelValidation: ...
    async def stream_answer(self, request: AnswerRequest): ...
    async def generate_structured(self, request, schema): ...
    async def embed(self, texts: list[str]) -> list[list[float]]: ...
```

## 6. 关键操作处理方案

### 6.1 Seed API 配置与模型验证

首版仅正式支持字节跳动 Seed 模型，通过火山方舟 Seed API 接入。

首版默认配置：

```text
数据面 Base URL：https://ark.cn-beijing.volces.com/api/v3
鉴权方式：Ark API Key
对话接口：`POST /chat/completions` 或 `POST /responses`
向量化接口：`POST /embeddings/multimodal`
```

配置流程：

1. 用户填写配置名称和 Ark API Key，普通设置中不要求填写 Base URL。
2. 系统加载应用内置的 Seed 模型目录，目录记录模型 ID、角色、上下文长度和支持的 API 类型。
3. 用户选择需要使用的对话模型、Embedding 模型和视觉模型。
4. 系统分别执行轻量调用，验证当前 Ark API Key 是否有权调用所选模型。
5. 系统展示验证成功、未开通、无权限、限流或调用失败等状态。
6. 保存模型选择和能力验证结果，定期或在调用失败后重新验证。

注意：

- 仅凭 Ark API Key 不能可靠自动枚举该 Key 可调用的全部模型，因此首版采用“内置 Seed 模型目录 + 实际调用验证”，不承诺自动列出全部模型。
- Ark API Key 可以限制可调用的 Model ID；用户需要先在火山方舟开通对应模型。
- 如未来需要通过管控面自动查询账户已开通模型，需要额外接入 Access Key / Secret Key 鉴权，首版不引入。
- Tool Calling、视觉和结构化输出需要分别探测。
- 未通过结构化输出测试的模型，应降级为普通 JSON Prompt 并执行解析校验。
- Ark API Key 使用 `safeStorage` 加密，不直接暴露给 React Renderer。

```text
┌────────────────────────────────────────────────────┐
│ 配置 Seed API                                     │
├────────────────────────────────────────────────────┤
│ 名称      [ 我的 Seed API                      ]   │
│ Ark Key   [ ************************************ ]   │
│                                                    │
│ [验证密钥并加载 Seed 模型]                         │
│                                                    │
│ Seed 模型             角色              状态         │
│ doubao-seed-...       对话 / 视觉       可调用       │
│ doubao-embedding-...  Embedding         可调用       │
│ doubao-seed-...       对话              未开通       │
└────────────────────────────────────────────────────┘
```

### 6.2 用户切换模型

不同角色的模型必须采用不同处理方式。

#### 切换对话或写作模型

- 已生成的历史消息保持不变。
- 当前正在生成的回答继续使用原模型，不能中途热切换。
- 新模型从下一条用户消息开始生效。
- 每条回答记录实际模型、参数、知识索引版本和生成时间。
- 全局默认模型变更只影响新建对话。
- 用户可以主动切换当前对话使用的模型。
- 当新模型上下文窗口较小时，使用“近期消息 + 历史摘要”，但不删除原始对话。

```text
┌──────────────────────────────────┐
│ 切换对话模型                     │
├──────────────────────────────────┤
│ 新模型：[ chat-pro-v2        ]   │
│                                  │
│ ● 仅切换当前对话                 │
│ ○ 设为新对话默认模型             │
│                                  │
│ 历史回答不会重新生成，新模型将在 │
│ 下一条消息开始使用。             │
│                                  │
│              [取消] [确认切换]   │
└──────────────────────────────────┘
```

#### 切换 Embedding 模型

不同 Embedding 模型的向量维度和语义空间通常不兼容，因此切换后必须重建索引。

处理流程：

```text
当前索引 V1 正常提供检索
        ↓
后台使用新 Embedding 构建 V2
        ↓
验证 V2 完整性与基本检索效果
        ↓
原子切换当前索引到 V2
        ↓
短期保留 V1，支持回滚
```

重建前应展示：

- 涉及的文档数和文本块数
- 预计 API 调用量和费用提示
- 当前索引仍可继续使用的说明
- 暂停、取消和失败重试入口

每个索引版本至少记录：

```text
embedding_provider
embedding_model
embedding_dimension
chunking_version
parser_version
created_at
status
```

#### 切换视觉或 OCR 模型

- 不自动重新处理历史文档。
- 用户可以针对单个文档或整个知识库发起重新解析。
- 重新解析会生成新的文档版本，旧版本短期保留。
- 如果重新解析改变了文本块，需要为新版本重新建立向量索引。

#### 切换重排序模型

- 不需要重建向量索引。
- 从下一次检索开始使用新模型。
- 应记录重排序模型版本，方便解释检索结果变化。

### 6.3 文档导入与索引

导入流程：

```text
选择文件或输入网页链接
        ↓
计算原始文件 Hash，检测完全重复
        ↓
解析文档结构或执行 OCR
        ↓
用户可检查解析结果
        ↓
标准化文本并检测内容重复
        ↓
按照文档结构分块
        ↓
写入 SQLite 元数据与 FTS5
        ↓
调用 Embedding 并写入 LanceDB
        ↓
标记文档版本为可检索
```

每个文本块至少保存：

```text
chunk_id
source_id
source_version
page_number
bounding_box
heading_path
original_text
normalized_text
content_hash
index_version
```

文档在完整索引完成前显示为“处理中”，不能悄悄出现在检索结果中。

### 6.4 重复文件检测

首版采用两级检测：

1. **原始文件 SHA-256**：判断文件内容是否完全一致。
2. **标准化正文 Hash**：判断不同文件格式或文件名下的正文是否一致。

发现重复时不应直接删除，应让用户选择：

- 跳过导入
- 保留为独立副本
- 关联到已有文档

近似重复检测可以在第二阶段使用分段 Hash 或语义相似度实现，但必须展示判断依据，避免误删。

### 6.5 网页更新检测

导入网页时保存：

```text
url
etag
last_modified
content_hash
content_snapshot
last_checked_at
```

检查流程：

1. 优先使用 `ETag` 和 `Last-Modified` 判断是否可能更新。
2. 重新提取正文并计算正文 Hash。
3. 正文变化时创建新版本，不覆盖旧版本。
4. 展示新旧版本差异，让用户决定是否更新当前知识索引。
5. 更新索引时，仅重新处理变化的文本块。

首版仅在应用运行期间检查网页更新，不安装系统常驻后台服务。

### 6.6 过期资料标记

“过期”不能完全交给模型自动决定。建议使用以下可解释状态：

- 网页已有新版本。
- 文档被用户指定的新文件替代。
- 超过用户设置的复查时间。
- 内容与较新资料发生冲突。
- 模型推测可能过时，等待用户确认。

模型推测只能生成待确认建议，不能直接将资料设为过期。

### 6.7 自动归类与自动关联

自动归类建议使用“规则优先、模型辅助”的方式：

1. 根据文件夹、文件名、标题、作者、时间等元数据初步分类。
2. 使用模型生成主题标签建议。
3. 用户确认或修改标签。
4. 保存用户修正结果，后续优先复用。

自动关联应保存关联类型与依据，例如：

```text
关联类型：相同主题 / 引用关系 / 相同人物 / 观点冲突 / 版本替代
关联依据：匹配文本块、元数据或模型解释
置信程度：高 / 中 / 低
确认状态：系统建议 / 用户确认 / 用户拒绝
```

首版避免构建复杂知识图谱，只需保存可查询的关联关系。

### 6.8 混合检索

推荐检索流程：

```text
用户问题
   ├── SQLite FTS5 关键词检索
   └── LanceDB 向量检索
              ↓
        合并与去重
              ↓
        可选重排序
              ↓
        组装引用上下文
              ↓
        模型生成结构化回答
              ↓
        CitationValidator 校验
```

关键词检索适合精确术语、人名和数字；向量检索适合语义相近内容。两者不应互相替代。

### 6.9 回答与引用校验

要求模型输出结构化结果：

```json
{
  "paragraphs": [
    {
      "text": "回答段落内容",
      "evidence_chunk_ids": ["chunk_123", "chunk_456"]
    }
  ],
  "insufficient_evidence": false
}
```

`CitationValidator` 必须验证：

- `chunk_id` 是否真实存在。
- `chunk_id` 是否来自本次检索候选。
- 引用对应的文档版本是否仍然有效。
- 页码、标题路径和原文是否能够定位。
- 回答段落是否至少具有一个有效证据。

验证失败时：

- 删除无效引用。
- 标记该段“证据不足”。
- 必要时触发一次重新检索与重新生成。
- 不允许向用户展示模型虚构的引用。

相关程度不建议仅显示“92%”这类容易误导的数字。推荐展示：

```text
证据强度：高
检索排名：第 1
语义相关度：0.82
关键词命中：4 项
```

### 6.10 后台任务与错误恢复

以下操作必须作为持久化后台任务：

- 文档解析与 OCR
- 文本分块
- Embedding 调用
- 索引构建和重建
- 网页更新检查
- 自动分类和关联

每个任务至少保存：

```text
job_id
job_type
target_id
status
progress
checkpoint
retry_count
error_message
created_at
updated_at
```

任务状态建议：

```text
pending → running → completed
                  ├→ paused
                  ├→ cancelled
                  └→ failed → retrying
```

应用重新启动后，`BackgroundJobManager` 应检查未完成任务并恢复处理。

## 7. 本地存储与安全

### 7.1 默认本地保存内容

- 原始文件与网页快照
- 文本解析和 OCR 结果
- 文档版本与文本块
- 全文索引与向量索引
- 对话历史和生成内容
- API 配置与模型能力信息
- 后台任务状态
- 自动分类与关联关系

### 7.2 安全要求

- React Renderer 不得直接访问 API Key。
- `nodeIntegration` 必须关闭。
- `contextIsolation` 必须开启。
- 只通过 Preload 暴露有限、明确的 IPC 接口。
- API Key 使用 Electron `safeStorage` 加密。
- 调用云端模型前，明确区分将要发送的用户问题、历史消息和知识库片段。
- 日志中不得记录 API Key 和完整敏感文档正文。

## 11. 关键风险与约束

| 风险 | 应对方案 |
|---|---|
| Seed 模型版本、能力和可用状态发生变化 | 维护内置模型目录、执行能力验证并提供降级策略 |
| 文档解析无法准确保留页码和结构 | 保存原文定位信息，并提供解析结果检查 |
| 模型生成虚假引用 | 结构化输出 + 后端 CitationValidator |
| 更换 Embedding 导致索引不可用 | 双版本后台重建和原子切换 |
| 桌面应用退出导致长任务丢失 | SQLite 持久化任务和检查点恢复 |
| 自动分类与过期判断误判 | 展示依据并要求用户确认 |
| 本地数据体积持续增加 | 文档版本清理、索引回收和空间占用展示 |
| API 调用费用不可控 | 调用前估算、批量限速和用量统计 |

## 12. 当前确定的架构决策

1. 桌面端首版使用 Electron、React、TypeScript 和 Vite。
2. 文档处理与 RAG 核心使用独立 Python Worker。
3. Electron Main 与 Python Worker 使用 JSON-RPC over stdio 通信。
4. SQLite 保存业务数据、对话、任务和全文索引。
5. LanceDB 保存向量索引。
6. 使用混合检索，不只依赖向量搜索。
7. 使用火山方舟 Ark SDK 调用 Seed 模型，使用 LangChain Core 处理 Prompt、结构化输出和流式流程封装。
8. 核心业务数据与 LangChain 解耦。
9. LangGraph 在第二阶段用于复杂写作工作流，不用于首版普通问答。
10. 首版仅正式支持字节跳动 Seed 模型，通过火山方舟 Seed API 和 Ark SDK 调用，不支持任意 OpenAI-compatible API，也不支持本地大语言模型。
11. 对话模型可以从下一条消息开始切换；Embedding 模型切换必须重建索引。
12. 所有引用必须经过后端校验后才能展示。
