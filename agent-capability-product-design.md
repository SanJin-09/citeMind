# citeMind Agent 能力：定位、发散与第一轮收缩

> 文档状态：第一轮产品与架构收缩草案  
> 更新日期：2026-06-14  
> 目标：明确为什么做 Agent、为谁做、先解决什么，以及 MCP、Skill、子 Agent 和 Agent Loop 各自应处于什么位置。

## 1. 先定义问题，而不是先定义 Agent

citeMind 当前已经完成“导入资料 -> 建立索引 -> 检索问答 -> 引用校验 -> 导出”的可信问答闭环。

当前用户仍需手动完成的工作是：

1. 将模糊目标拆成多个研究问题。
2. 反复检索、阅读、比较来源并发现证据缺口。
3. 将零散回答组织成大纲或交付物。
4. 检查每个结论是否有证据、来源是否冲突。
5. 在失败、信息不足或需要外部资料时决定下一步。

因此，Agent 能力要解决的核心问题不是“让模型能够调用工具”，而是：

> 用户给出一个交付目标和资料边界，citeMind 能持续执行多步研究与写作任务，最终交付一个证据可追溯、过程可审计、随时可干预的成果。

## 2. 精确产品定位

### 2.1 推荐定位

**citeMind 是面向重度资料工作者的可审计研究与写作 Agent。**

它帮助用户把个人知识库和经过确认的外部资料，转化为研究简报、资料对比、复习提纲和长文初稿；所有事实性结论都能追溯到真实来源。

### 2.2 首要目标用户

第一优先建议选择：

**需要基于大量资料持续产出报告、文章或研究结论的个人知识工作者。**

典型用户包括咨询顾问、研究人员、分析师和深度内容创作者。学生仍然适配，但不作为 Agent 阶段的唯一首要用户。

选择原因：

- 他们有明确的“交付物”，Agent 是否成功容易判断。
- 他们需要跨来源比较、证据审计和长流程执行，现有问答无法完整覆盖。
- “可信引用”对他们具有直接价值，而不是附加功能。

### 2.3 一句话承诺

> 给 citeMind 一个研究或写作目标，它会在你允许的资料范围内规划、执行、检查并交付一份每个关键结论都有依据的成果。

### 2.4 明确不做

- 不做可以随意操作电脑和第三方服务的通用助手。
- 不做以“连接了多少 MCP Server”为卖点的 MCP 客户端。
- 不做无限循环、无限委派、无法预测费用的自主 Agent。
- 不让外部工具返回的内容绕过导入、快照和引用校验直接成为证据。
- 第一阶段不做 Skill 市场、Agent 市场和用户自定义代码执行。

## 3. 统一概念，避免能力重叠

| 概念 | 在 citeMind 中的定义 | 用户是否直接感知 |
|---|---|---|
| Tool | 单次、原子的受控能力，例如检索知识库、读取来源、校验引用 | 可在执行记录中查看 |
| MCP Connection | citeMind 与外部数据或工具服务的标准连接方式 | 在连接与授权页面感知 |
| Skill | 可版本化、可复用的任务配方，定义输入、步骤、可用工具、约束和输出格式 | 以“研究简报”等任务模板感知 |
| Sub-agent | 在有限上下文和权限内完成一个明确子任务的执行角色 | 默认弱感知，可查看委派记录 |
| Agent Run | 围绕一个用户目标持续执行、可暂停恢复的任务实例 | 主要交互对象 |
| Artifact | Agent 最终或中间交付的成果，例如大纲、对比表、研究报告 | 主要价值载体 |

关键关系：

```text
用户目标
  -> 选择 Skill
  -> 创建 Agent Run
  -> Agent Loop 调用原生 Tool / MCP Tool
  -> 必要时委派受限 Sub-agent
  -> 生成并校验 Artifact
```

## 4. 场景发散

| 候选场景 | 用户价值 | 与现有能力匹配 | 风险与成本 | 建议 |
|---|---:|---:|---:|---|
| 证据研究简报 | 高 | 高 | 中 | 首发主场景 |
| 多来源观点对比 | 高 | 很高 | 低 | 首发内置 Skill |
| 引用与冲突审计 | 高 | 很高 | 低 | 首发内置 Skill |
| 复习提纲与测验 | 中 | 高 | 低 | 后续快速扩展 |
| 大纲到长文初稿 | 高 | 高 | 中高 | 主流程稳定后加入 |
| 自动分类与知识库维护 | 中 | 中高 | 中 | 作为后台建议能力 |
| 自动搜索并导入外部资料 | 高 | 中 | 高 | 仅在审批后执行 |
| 发布内容、发送邮件、修改第三方数据 | 中 | 低 | 很高 | 暂不实现 |
| 通用聊天 Agent | 低 | 低 | 很高 | 明确放弃 |

## 5. 第一轮收缩：Agent MVP

### 5.1 MVP 核心任务

首发只完整打通一个 Agent Loop：

> 根据用户选择的知识库资料和交付要求，生成一份证据研究简报。

它应完成：

1. 理解交付目标、受众、篇幅和资料范围。
2. 生成可编辑的执行计划和费用上限。
3. 拆分研究问题并检索证据。
4. 识别证据缺口和来源冲突。
5. 在需要扩大资料范围时请求用户批准。
6. 生成结构化成果。
7. 执行引用校验和冲突审计。
8. 保存成果、执行过程和失败原因。

### 5.2 首发内置 Skill

三个 Skill 共享同一套 Agent Runtime 和工具，不建立三套独立流程：

1. **证据研究简报**：围绕目标形成结构化研究结论。
2. **多来源观点对比**：按议题比较来源共识、差异和证据强弱。
3. **引用与冲突审计**：检查已有文本中的无证据结论、错误引用和来源冲突。

Skill 不应只是 Prompt 文件。最小定义应包含：

```yaml
id: research-brief
version: 1
inputs:
  - goal
  - audience
  - output_length
  - source_scope
allowed_tools:
  - knowledge.search
  - knowledge.read_chunk
  - knowledge.inspect_source
  - evidence.validate
  - artifact.save
policy:
  max_steps: 16
  max_model_calls: 12
  external_read: ask
  external_write: deny
output_schema: research_brief_v1
```

### 5.3 首发原生 Tools

- `knowledge.search`：在指定知识库、来源范围和索引版本内检索。
- `knowledge.read_chunk`：读取候选证据及其定位信息。
- `knowledge.inspect_source`：读取来源元数据、版本和处理状态。
- `evidence.validate`：校验成果段落与证据关系。
- `artifact.save`：保存草稿或最终成果。
- `user.request_input`：在缺少关键约束或需要批准时暂停并询问用户。

现有 `HybridRetrievalService`、`CitationValidator`、来源服务和模型网关应直接适配成原生 Tool。不要为了“统一”而让 citeMind 内部核心能力先绕一层 MCP。

## 6. MCP 的正确位置

### 6.1 优先级

1. **citeMind 作为 MCP Host/Client**：连接外部只读数据和发现工具。
2. **citeMind 核心服务使用原生 Tool Adapter**：保持类型、安全与业务约束。
3. **citeMind 作为 MCP Server**：后续向其他 Agent 暴露只读知识库检索能力。

### 6.2 首发 MCP 边界

- 只支持经过用户显式配置的连接。
- 首发只开放外部读取，不开放外部写入和破坏性操作。
- 外部 MCP 返回内容只能作为“候选资料”。
- 候选资料必须经过用户确认、保存快照、导入和索引后，才能成为最终成果的可引用证据。
- 外部 Tool 描述和注解视为不可信提示，权限决策必须由 citeMind 自己的策略决定。

```text
外部 MCP 搜索结果
  -> 候选资料
  -> 用户确认
  -> citeMind 导入 / 快照 / 索引
  -> 检索候选
  -> CitationValidator
  -> 可展示引用
```

### 6.3 暂不依赖 MCP Tasks

MCP `Tasks` 适合长任务，但在当前规范中仍属于实验能力。citeMind 已有持久化后台任务，Agent Run 的恢复不应依赖 MCP Tasks；未来可在连接层做兼容。

## 7. Agent Loop 与子 Agent

### 7.1 采用有界循环

```text
理解目标
  -> 生成计划
  -> 用户确认
  -> 执行下一步
  -> 检查结果与剩余预算
  -> 继续 / 请求用户输入 / 结束
  -> 引用与冲突校验
  -> 交付 Artifact
```

每次 Run 必须有：

- 最大步骤数、模型调用数、Token、费用和运行时间。
- 明确的资料范围与索引版本。
- 可暂停、取消、恢复的状态。
- 可见的计划、工具调用、证据和失败原因。
- 完成条件和停止原因。

### 7.2 子 Agent 的首发边界

子 Agent 是“受限任务执行器”，不是独立人格。

建议内置角色：

- `Evidence Scout`：针对一个研究问题查找候选证据。
- `Analyst`：比较证据、识别共识与冲突。
- `Writer`：依据已确认结论生成成果。
- `Auditor`：独立执行引用和冲突检查。

MVP 约束：

- 最多并行两个 Evidence Scout。
- 子 Agent 不得继续创建子 Agent。
- 子 Agent 只获得完成子任务所需的最小上下文和工具。
- Writer 不直接搜索外部资料，Auditor 不修改原始证据。

### 7.3 是否立即引入 LangGraph

当前代码没有实际依赖 LangChain 或 LangGraph，普通可信问答由显式 Python 服务完成。

MVP 建议继续使用显式状态机实现有界 Agent Loop，原因是更容易调试、审计和控制权限。等到多个 Skill 出现明显的分支、并行和恢复重复逻辑后，再评估引入 LangGraph，避免仅因为使用了“Agent”概念而增加框架依赖。

## 8. 权限与信任模型

| 操作级别 | 示例 | 默认策略 |
|---|---|---|
| 本地只读 | 检索知识库、读取来源元数据 | 自动允许并记录 |
| 本地生成 | 保存草稿和成果 | Run 创建时一次授权 |
| 外部只读 | MCP 搜索、读取网页或第三方资料 | 首次调用前确认范围 |
| 导入资料 | 保存外部内容并加入知识库 | 展示候选后确认 |
| 外部写入 | 发布、发送、修改第三方数据 | MVP 禁止 |
| 破坏性操作 | 删除资料、覆盖成果 | 每次明确确认 |

Agent 的每个事实性输出都必须区分：

- 已验证证据支持的结论。
- 存在冲突的结论。
- 模型推断。
- 证据不足或待用户确认的内容。

## 9. UI/UX 草案

现有三栏结构可以自然演进，不需要新建完全独立的 Agent 产品界面。

```text
┌──────────────────────────────────────────────────────────────────────────┐
│ 当前知识库                  [对话] [研究任务]              连接  设置    │
├──────────────────┬────────────────────────────────┬──────────────────────┤
│ 资料范围         │ 研究任务：竞品方案评估          │ 执行与证据           │
│                  │                                │                      │
│ ☑ 产品方案.pdf   │ Skill：证据研究简报            │ 计划  6/9            │
│ ☑ 访谈记录.docx  │ 目标：形成面向决策者的简报      │ ✓ 检索市场证据       │
│ ☐ 历史草稿.md    │                                │ ✓ 对比两种方案       │
│                  │ [查看/编辑计划] [暂停] [取消]   │ ● 检查来源冲突       │
│ + 添加资料       │                                │ ○ 生成最终简报       │
│ + 外部资料候选   │ ┌────────────────────────────┐ │                      │
│                  │ │ 当前成果预览               │ │ 待确认               │
│                  │ │ 1. 核心结论…… [1][2]       │ │ 是否导入外部资料 A？ │
│                  │ │ 2. 风险与分歧…… [3]        │ │ [拒绝] [查看] [允许] │
│                  │ └────────────────────────────┘ │                      │
│                  │                                │ 证据 / 调用记录      │
└──────────────────┴────────────────────────────────┴──────────────────────┘
```

核心交互原则：

- 用户首先选择“要交付什么”，而不是选择“要启动哪个 Agent”。
- 执行计划默认可见、可编辑。
- Agent 需要用户决定时进入明确的等待状态，不在聊天文本里埋确认问题。
- 工具调用记录服务于解释和审计，不把原始技术日志直接倾倒给用户。

## 10. 最小架构改动

```text
React Renderer
  -> Agent Application Service
      -> Agent Orchestrator（显式状态机）
          -> Skill Registry
          -> Policy / Budget Guard
          -> Native Tool Registry
          -> MCP Client Manager
          -> ModelGateway
          -> CitationValidator
```

为避免实体膨胀，MVP 建议只增加三个核心存储实体：

| 实体 | 作用 |
|---|---|
| AgentRun | 保存目标、Skill 版本、状态、预算、资料范围和索引版本 |
| AgentEvent | 追加式保存计划变化、工具调用、用户批准、错误与状态迁移 |
| Artifact | 保存中间成果和最终成果及其引用关系 |

`BackgroundJob` 继续表示导入、OCR、Embedding 和索引等基础设施任务；不要用它代替用户可感知的 Agent Run。

## 11. 验收指标

Agent MVP 不能只用“能跑起来”验收。

核心指标：

- 研究简报完成率。
- 用户无需手动重写即可接受的成果比例。
- 事实性结论有效引用覆盖率，目标为 100%。
- 引用校验失败后仍被展示的比例，目标为 0。
- 每个成果平均需要的用户干预次数。
- 单次 Run 的模型调用量、耗时和估算费用。
- 暂停、异常退出后恢复成功率。

固定评测集至少覆盖：

- 证据充分且无冲突。
- 多来源存在明显冲突。
- 证据不足，需要主动停止。
- 外部资料候选不能直接作为引用。
- 工具失败、模型输出格式错误和用户拒绝授权。

## 12. 建议实施顺序

### 阶段 A：证明 Agent 对用户有价值

- 实现 AgentRun、AgentEvent、Artifact。
- 将现有检索、来源读取和引用校验封装为原生 Tools。
- 实现“证据研究简报”有界循环。
- 提供计划确认、运行轨迹、暂停恢复和成果预览。
- 建立固定 Agent 评测集。

### 阶段 B：形成可复用 Skill Runtime

- 增加“多来源观点对比”和“引用与冲突审计”。
- 抽取版本化 Skill 定义、预算和权限策略。
- 加入受限 Evidence Scout 与 Auditor。

### 阶段 C：接入 MCP

- 实现 MCP Client Manager、连接配置和权限展示。
- 先接一个只读外部资料发现服务。
- 实现“候选资料 -> 用户确认 -> 导入索引 -> 可引用”的闭环。
- 后续再考虑将 citeMind 的知识库检索能力作为只读 MCP Server 暴露。

## 13. 第一轮决策结论

当前建议立即冻结的决策：

1. Agent 阶段定位为“可审计研究与写作”，不是通用 Agent 平台。
2. 首发主流程是“证据研究简报”，另外两个 Skill 复用同一 Runtime。
3. 核心知识库能力使用原生 Tool，MCP 用于外部连接。
4. 外部内容必须导入、快照和索引后才能成为证据。
5. Agent Loop 必须有界、可暂停、可恢复、可审计。
6. 子 Agent 仅做受限委派，不允许递归委派。
7. 首发禁止外部写入和破坏性 MCP Tool。
8. 暂不因 Agent 概念立即引入 LangGraph。

下一轮需要重点验证的产品假设：

1. 首要用户是否确认为咨询顾问、研究人员、分析师和深度内容创作者。
2. “研究简报”是否是最值得优先打通的交付物。
3. 用户希望 Agent 默认只使用当前知识库，还是默认主动建议寻找外部资料。
4. 成果主要以 Markdown 文档、结构化卡片，还是可继续编辑的写作页面呈现。
